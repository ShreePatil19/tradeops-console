# Invoice PDF fixtures

Five deterministic PDFs that exercise the kinds of variation a real ops desk sees. Generated by `evals/invoice/fixtures/build.ts` and committed to the repo so eval runs are reproducible across machines and CI.

## Fixtures

| File | Variant | Purpose |
|---|---|---|
| `01-clean.pdf` | clean | Standard 3-line invoice, all fields present. Baseline. |
| `02-partial.pdf` | partial | Missing invoice number, fewer line items. Tests fallback behaviour. |
| `03-stamped.pdf` | stamped | Red "RECEIVED" stamp overlaid at 12 degrees, semi-transparent, over the header. |
| `04-handwritten.pdf` | handwritten | Italic "short by 1 pallet" note overlaid in blue at the bottom. |
| `05-multipage.pdf` | multipage | 18 line items split across 2 pages with a continued header on page 2. |

## Regenerating

```bash
pnpm fixtures:invoice           # rebuild all 5 PDFs into this directory
pnpm fixtures:invoice -- --check  # verify the on-disk PDFs still match generator output
```

The output is byte-deterministic: creation and modification dates are pinned to `2024-01-01T00:00:00Z`, the producer string is set explicitly, and `pdf-lib`'s save mode uses `useObjectStreams: false` so object IDs are stable. CI can use `--check` to assert no drift.

## Tests

`tests/evals/invoice-fixtures.test.ts` covers:

- PDF magic header + reload via `PDFDocument.load`
- Page count per variant
- Supplier + invoice-number embedding in PDF metadata
- Variant-specific keyword tags (`stamped`, `handwritten`, `partial`)
- Byte determinism across two calls
- `FIXTURE_SPECS` shape: five named specs, every variant covered
- `buildAllFixtures()` returns a populated `Map<name, bytes>`

## Wiring into the live eval flow

Not yet. The current `/api/agents/invoice` route accepts JSON `messages` only. To run these PDFs end-to-end against the live API:

1. Extend the route to accept `Content-Type: multipart/form-data` with a `file` field containing the PDF binary.
2. Convert the file to a Gemini `FileData` or inline base64 attachment in the model call.
3. Update `evals/client.ts` to detect cases with a `fixture` field and POST as multipart instead of JSON.
4. Add new PDF-based cases to `cases.json` with `fixture: "01-clean.pdf"` etc.

That work is intentionally split into a follow-up PR. This PR ships only the fixtures so the generator and the artefacts can land independently of the route change.
